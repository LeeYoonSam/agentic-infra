# Ralph 사용 가이드

## Ralph란?

Ralph는 **oh-my-claudecode(OMC)** 플러그인의 핵심 스킬로, **PRD(Product Requirements Document) 기반 반복 실행 + 아키텍트 검증 루프**입니다.

일반적인 Claude Code 작업은 "한 번 실행하고 끝"이지만, Ralph는 다릅니다:

- 작업을 **User Story** 단위로 분해합니다
- 각 스토리마다 **수용 기준(Acceptance Criteria)**을 정의합니다
- 스토리별로 **구현 → 검증 → 완료** 사이클을 반복합니다
- 모든 스토리가 완료되면 **아키텍트 에이전트가 최종 검증**합니다
- 검증 실패 시 자동으로 수정하고 재검증합니다

즉, **"끝날 때까지 끝나지 않는"** 작업 보장 루프입니다.

## 언제 사용하나요?

| 상황 | 적합한 스킬 |
|------|-------------|
| 반드시 완료되어야 하는 복잡한 작업 | **`/ralph`** |
| 아이디어부터 코드까지 전체 자동화 | `/autopilot` |
| 계획을 먼저 세우고 싶을 때 | `/plan` |
| 단순한 한 줄 수정 | 직접 요청 |
| 병렬 실행만 필요할 때 | `/ultrawork` |

## 기본 사용법

### 1단계: Ralph 실행

Claude Code 프롬프트에서 `/ralph` 뒤에 작업 설명을 입력합니다:

```
/ralph 로그인 API에 OAuth2 소셜 로그인 기능을 추가해줘
```

```
/ralph 전체 테스트 커버리지를 80% 이상으로 올려줘
```

```
/ralph README.md를 한국어로 종합 작성해줘
```

### 2단계: PRD 자동 생성

Ralph가 실행되면 자동으로 `.omc/prd.json` 파일을 생성합니다. 이 파일에는:

- 작업 제목과 설명
- User Story 목록 (각각 고유 ID 부여)
- 스토리별 수용 기준 (Acceptance Criteria)
- 우선순위와 완료 상태

Claude가 PRD를 자동 생성한 후, **구체적이고 검증 가능한 수용 기준**으로 개선합니다.

### 3단계: 자동 실행 관찰

Ralph가 알아서 작업합니다:

1. **스토리 선택** — 가장 높은 우선순위의 미완료 스토리를 선택
2. **구현** — 전문 에이전트에게 작업 위임 (병렬 실행)
3. **검증** — 수용 기준을 하나씩 확인 (테스트, 빌드, 린트 실행)
4. **완료 표시** — 모든 기준 통과 시 `passes: true`로 변경
5. **다음 스토리** — 미완료 스토리가 남아있으면 1번으로 돌아감

### 4단계: 아키텍트 검증

모든 스토리가 완료되면 아키텍트 에이전트가 최종 검증합니다:

- **소규모 변경** (파일 5개 미만, 100줄 미만): Sonnet 급 검증
- **일반 변경**: Sonnet 급 검증
- **대규모/보안 관련** (파일 20개 초과): Opus 급 정밀 검증

검증 통과 시 자동으로 `/cancel`을 실행하여 깨끗하게 종료합니다.

### 5단계: 완료

작업이 끝나면 다음이 생성/업데이트됩니다:
- `.omc/prd.json` — 모든 스토리가 `passes: true`
- `.omc/progress.txt` — 구현 내역, 변경 파일, 학습 내용 기록

## PRD 없이 사용하기 (`--no-prd`)

간단한 수정이라 PRD가 과할 때는 `--no-prd` 플래그를 사용합니다:

```
/ralph --no-prd 오타 수정하고 빌드 확인해줘
```

이 모드에서는:
- `prd.json` 생성을 건너뜁니다
- 스토리 추적 없이 작업합니다
- 일반적인 검증만 수행합니다

:::tip
`--no-prd` 플래그는 프롬프트 어디에나 넣을 수 있습니다:
```
/ralph 오타 수정 --no-prd
/ralph --no-prd 오타 수정
```
:::

## 고급: ralph-init으로 PRD 먼저 만들기

복잡한 작업은 PRD를 먼저 검토한 후 실행하고 싶을 수 있습니다:

### 방법 1: ralph-init 사용

```
/ralph-init OAuth2 소셜 로그인 기능 추가
```

이 명령은:
1. 요구사항을 분석하여 PRD를 생성합니다
2. `.omc/plans/prd-{slug}.md`에 저장합니다
3. **실행하지 않고 멈춥니다** — PRD만 만들고 대기

PRD를 검토/수정한 후 ralph를 실행합니다:
```
/ralph OAuth2 소셜 로그인 기능 추가
```

### 방법 2: prd.json 직접 작성

`prd.json`을 프로젝트 루트 또는 `.omc/` 폴더에 직접 만들 수도 있습니다:

```json
{
  "title": "OAuth2 소셜 로그인",
  "description": "Google, GitHub 소셜 로그인 지원 추가",
  "stories": [
    {
      "id": "US-001",
      "title": "Google OAuth2 연동",
      "priority": 1,
      "passes": false,
      "acceptanceCriteria": [
        "Google OAuth2 클라이언트 설정 코드가 존재함",
        "/auth/google 엔드포인트가 동작함",
        "콜백에서 JWT 토큰을 발급함",
        "관련 테스트가 통과함"
      ]
    },
    {
      "id": "US-002",
      "title": "GitHub OAuth 연동",
      "priority": 2,
      "passes": false,
      "acceptanceCriteria": [
        "/auth/github 엔드포인트가 동작함",
        "GitHub 프로필에서 이메일을 가져옴",
        "관련 테스트가 통과함"
      ]
    }
  ]
}
```

그 다음 Ralph를 실행하면 이 PRD를 자동으로 인식합니다:
```
/ralph OAuth2 소셜 로그인 구현해줘
```

## prd.json 구조 상세

```json
{
  "title": "string — 작업 제목",
  "description": "string — 전체 작업 설명",
  "stories": [
    {
      "id": "US-001",           // 고유 식별자
      "title": "스토리 제목",     // 짧은 제목
      "description": "상세 설명", // (선택) 상세 설명
      "priority": 1,             // 우선순위 (1이 가장 높음)
      "passes": false,           // 완료 여부 (Ralph가 자동 업데이트)
      "acceptanceCriteria": [    // 수용 기준 목록
        "구체적이고 검증 가능한 기준 1",
        "구체적이고 검증 가능한 기준 2"
      ],
      "notes": "구현 메모"       // (선택) Ralph가 추가하는 메모
    }
  ]
}
```

### 좋은 수용 기준 vs 나쁜 수용 기준

| 나쁜 예 (너무 모호함) | 좋은 예 (구체적, 검증 가능) |
|----------------------|---------------------------|
| "구현이 완료됨" | "src/auth/google.ts 파일이 존재함" |
| "코드가 잘 동작함" | "/auth/google GET 요청 시 302 리다이렉트 응답" |
| "테스트 통과" | "pnpm test 실행 시 auth 관련 테스트 5개 이상 통과" |
| "에러 없음" | "pnpm build 실행 시 exit code 0" |

## 실행 흐름 다이어그램

```
/ralph <작업 설명>
        │
        ▼
┌─────────────────┐
│ 1. PRD 생성/로드  │ ← prd.json 자동 생성 또는 기존 파일 로드
└────────┬────────┘
         ▼
┌─────────────────┐
│ 2. 스토리 선택    │ ← 가장 높은 우선순위의 passes:false 스토리
└────────┬────────┘
         ▼
┌─────────────────┐
│ 3. 구현          │ ← 전문 에이전트에게 위임 (병렬 실행)
└────────┬────────┘
         ▼
┌─────────────────┐
│ 4. 검증          │ ← 수용 기준 하나씩 확인
└────────┬────────┘
         │
    통과? ├── No ──→ 3번으로 돌아가 수정
         │
         ▼ Yes
┌─────────────────┐
│ 5. 스토리 완료    │ ← passes: true 표시
└────────┬────────┘
         │
  남은 스토리? ├── Yes ──→ 2번으로 돌아감
         │
         ▼ No
┌─────────────────┐
│ 6. 아키텍트 검증  │ ← 최종 품질 검증
└────────┬────────┘
         │
    통과? ├── No ──→ 수정 후 재검증
         │
         ▼ Yes
┌─────────────────┐
│ 7. 완료 & 정리   │ ← /cancel로 상태 정리
└─────────────────┘
```

## 관련 명령어

| 명령어 | 설명 |
|--------|------|
| `/ralph <작업>` | Ralph 루프 시작 |
| `/ralph --no-prd <작업>` | PRD 없이 간단 모드로 실행 |
| `/ralph-init <작업>` | PRD만 생성 (실행하지 않음) |
| `/cancel` | 진행 중인 Ralph 루프 중지 및 상태 정리 |

## FAQ 및 주의사항

### Q: Ralph가 무한 루프에 빠지면?

`/cancel`을 입력하면 즉시 중지됩니다. "stop", "cancel", "abort" 같은 단어를 입력해도 됩니다.

### Q: 이전 Ralph 작업의 prd.json이 남아있으면?

Ralph는 기존 `prd.json`을 자동으로 인식합니다. 새 작업을 시작하려면:
- `.omc/prd.json`을 삭제하거나
- 새 내용으로 덮어쓰면 됩니다

### Q: 중간에 Claude Code를 종료하면?

Ralph는 `.omc/` 폴더에 상태를 저장합니다:
- `prd.json` — 스토리 진행 상태
- `progress.txt` — 구현 기록
- `state/` — 세션 상태

다음 대화에서 `/ralph`를 다시 실행하면 이어서 작업할 수 있습니다.

### Q: 수용 기준을 중간에 바꾸고 싶으면?

`.omc/prd.json`을 직접 편집하면 됩니다. Ralph는 매 반복마다 `prd.json`을 새로 읽으므로 변경사항이 즉시 반영됩니다.

### Q: Ralph vs Autopilot 차이는?

- **Ralph**: 기존 코드베이스에서 **구체적인 작업**을 확실하게 완료 (PRD 기반)
- **Autopilot**: **아이디어에서 동작하는 코드까지** 전체 파이프라인 자동화

### 주의사항

:::caution
- Ralph는 **복잡한 작업용**입니다. 한 줄 수정에는 과합니다.
- 첫 PRD의 수용 기준이 구체적일수록 결과가 좋습니다.
- `progress.txt`는 삭제하지 마세요 — 반복 간 학습 내용이 담겨 있습니다.
- Ralph 실행 중에는 다른 persistent mode 스킬 (ultraqa 등)을 동시에 사용하지 마세요.
:::
